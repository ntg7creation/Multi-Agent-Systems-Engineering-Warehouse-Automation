2.  Environment  Specification  
2.1  Environment  Model  
The  warehouse  environment  shall  be  represented  as  a  discrete  grid  of  cells.  Each  
cell
 
shall
 
belong
 
to
 
one
 
of
 
the
 
following
 
categories:
 
●  Free  traversal  cell.  ●  Pickup  location.  ●  Delivery  location.  ●  A  blocked  cell  such  as  a  shelf  or  obstacle.  
 
2.2  Environment  Taxonomy  
The  system  environment  is  classified  as  follows:  
●  Cooperative  -  all  agents  pursue  the  shared  goal  of  completing  delivery  tasks.  ●  Discrete  -  time  progresses  in  turns  or  simulation  steps.  ●  Partially  observable  -  each  agent  can  perceive  only  a  limited  local  
neighbourhood.
 ●  Dynamic  -  the  state  of  the  environment  may  change  due  to  agent  movement,  
task
 
completion,
 
and
 
newly
 
blocked
 
or
 
occupied
 
cells.
 ●  Constrained  -  no  two  agents  may  occupy  the  same  cell  at  the  same  time.   
2.3  Physical  Boundaries  
The  physical  boundaries  of  the  system  are  the  dimensions  of  the  warehouse  grid.  
Agents
 
may
 
not
 
move
 
outside
 
the
 
valid
 
grid
 
area.

The  logical  boundaries  of  the  system  include:  
●  The  set  of  active  tasks  currently  available.  ●  Each  agent’s  local  perception  radius,  ●  Each  agent’s  internal  memory  or  explored  map,

2.5  Environmental  Constraints  
The  environment  shall  enforce  the  following  constraints:  
●  Blocked  cells  are  not  traversable.  ●  An  agent  carrying  an  item  may  not  pick  up  another  item.  ●  Each  item  must  be  associated  with  a  valid  destination.  ●  Simultaneous  occupancy  of  a  cell  by  multiple  agents  is  forbidden.  ●  Agents  may  need  to  replan  paths  when  local  conditions  change.  
 
 
3.  Agent  Specification   
3.1  Agent  Type  
The  system  shall  initially  include  one  agent  type:  Warehouse  Transport  Agent.  
3.2  Agent  Goals  
Each  warehouse  transport  agent  shall  pursue  the  following  individual  goal  sequence:  
1.  Receive  a  transport  task.  2.  Travel  to  a  valid  cell  adjacent  to  the  pickup  location.  3.  Pick  up  the  assigned  item  from  the  adjacent  pickup  cell.  4.  Navigate  to  the  item’s  destination.  5.  Place  the  item  at  the  destination.  6.  Become  available  for  the  next  task.   
3.3  Perception  Capabilities  
Each  agent  shall  be  able  to  perceive:  
●  Nearby  free  and  blocked  cells  within  its  perception  radius.  ●  Nearby  agents.  ●  Nearby  pickup  and  delivery  points  if  visible.      Shenkar  -  Software  Requirements  Specification

3.4  Action  Capabilities  
Each  agent  shall  support  the  following  actions:  
●  Move  one  cell  in  a  valid  direction.  ●  Pick  up  an  item  from  a  delivery  point.  ●  Place  an  item  at  an  adjacent  cell.  ●  Wait  for  one  simulation  step.  ●  Pick  up  an  item  from  an  adjacent  pickup  cell

3.6  Internal  State 
Each  agent  shall  maintain  an  internal  state  including:  
●  Current  position.  ●  Availability  status.  ●  Carrying  status.  ●  Assigned  task  identifier.  ●  Partial  internal  map  of  explored  areas.  ●  Current  route  or  target  cell.  
 
3.7  Agent  Limitations  
●  An  agent  may  perform  at  most  one  action  per  turn.  ●  An  agent  may  carry  only  one  item  at  a  time.  ●  An  agent  has  only  local  perception  and  does  not  initially  know  the  full  global  
state.
 ●  An  agent  must  obey  collision  and  occupancy  constraints.   
3.8  Agent  Path  Planning  and  Route  Selection  
Each  agent  shall  use  a  non-deterministic,  adaptive  path-planning  strategy  rather  
than
 
always
 
selecting
 
the
 
single
 
shortest
 
path.
 
The  agent  shall  attempt  to  move  directly  toward  its  current  target  when  the  route  is  
clear
 
and
 
valid.
 
However,

the
 
agent
 
detects
 
a
 
blocked

obstacle,

occupied

Route  selection  shall  be  based  on  weighted  probabilities  rather  than  a  fully

●  estimated  distance  to  the  target      Shenkar  -  Software  Requirements  Specification

4.  Functional  Requirements   
4.1  System-Level  Functional  Requirements  
Functional  Requirement  
Functional  Requirement  Description  
FR-01  
The  system  shall  initialize  a  warehouse  grid  with  valid  cell  types,  obstacles,  pickup  points,  delivery  points,  items,  and  agents   
FR-02  
The  system  shall  maintain  a  task  list  of  active  pickup-and-delivery  jobs   
FR-02a  
The  system  shall  generate  pickup-and-delivery  tasks  according  to  a  predefined  or  seed-based  deterministic  process.  Tasks  shall  be  produced  as  a  sequence  (queue)  of  orders,  where  each  order  specifies:  a  pickup  location  (one  of  the  defined  pickup  cells),  a  corresponding  delivery  location,  and  an  item  identifier     
FR-02b  
The  system  shall  represent  each  item  as  an  entity  with  a  unique  item  ID  and  a  spatial  location  within  the  warehouse  grid   
FR-03  The  system  shall  execute  the  simulation  in  discrete  time  steps   
FR-04  
The  system  shall  prevent  illegal  moves,  including  movement  into  blocked  cells,  out-of-bounds  cells,  and  currently  occupied  cells   
FR-05  
The  system  shall  update  the  global  warehouse  state  after  each  simulation  step   
FR-06  
The  system  shall  detect  task  completion  when  an  assigned  item  is  successfully  delivered  to  its  destination   
FR-07  
The  system  shall  record  performance  statistics  for  all  agents  and  for  the  system  as  a  whole    
4.2  Agent-Level  Functional  Requirements  
Functional  Requirement  
Functional  Requirement  Description  
FR-08  An  idle  agent  shall  be  able  to  select  or  receive  an  available  task  
FR-09  
An  agent  shall  compute  a  path  toward  its  current  target  using  available  local  and  stored  information  
    Shenkar  -  Software  Requirements  Specification

FR-10  
An  agent  shall  replan  or  wait  when  its  next  move  becomes  invalid  due  to  dynamic  changes  
FR-11  
An  agent  shall  pick  up  an  item  only  when  the  agent  is  located  in  a  valid  free  cell  adjacent  to  the  item’s  pickup  cell,  is  not  already  carrying  another  item,  and  the  item  is  assigned  to  that  agent.  The  agent  shall  not  occupy  the  pickup/item  cell  itself  when  performing  the  pickup  action.   
FR-12  
An  agent  shall  place  an  item  only  when  the  agent  is  located  in  a  valid  free  cell  adjacent  to  the  item’s  destination  cell,  and  only  if  the  carried  item  corresponds  to  that  destination.  The  agent  shall  not  occupy  the  destination  cell  itself  when  performing  the  placement  action.  
FR-13  
An  agent  shall  update  its  internal  state  after  movement,  pickup,  placement,  or  waiting  
FR-14  An  agent  shall  avoid  collisions  by  not  entering  occupied  cells

FR-17

6.3  Extended  Coordination  Mechanism  
The  system  shall  be  extensible  to  auction-based  task  allocation.  In  this  model:  
●  A  new  task  is  announced.  ●  Eligible  agents  compute  a  bid.  ●  The  bid  may  depend  on  distance,  expected  cost,  or  current  availability.  ●  The  task  is  assigned  to  the  most  suitable  bidder.

7.4  Global  System  Success  Metrics  
The  system  shall  be  evaluated  using  the  following  metrics:   Completion  Time:  total  simulation  steps  required  to  complete  all  deliveries.  
 
Task  Completion  Rate  (TCR):  
TCR  =  number  of  completed  deliveries  /  total  number  of  delivery  tasks  
 
Agent  Efficiency  (AEi):  
AEi  =  useful  actions  of  agent  i  /  total  actions  of  agent  i  
 
Collision  Avoidance  Score  (CAS):  
CAS  =  1  -  (number  of  collision  violations  /  total  movement  attempts)  
 
Throughput  (TH):  
TH  =  completed  deliveries  /  simulation  time

8.  Validation  Scenarios  
 №  Scenario  Description  Purpose  
1  
Single-Agent  Baseline  
A  single  agent  operates  in  a  small  warehouse  with  no  dynamic  obstacles  and  a  fixed  set  of  delivery  tasks  
Validate  the  correctness  of  basic  navigation,  pickup,  and  delivery  behavior   
2  
Multi-Agent  Cooperative  Delivery  
Several  agents  operate  simultaneously  in  the  same  warehouse  with  multiple  pending  tasks  
Validate  concurrent  execution,  task  distribution,  and  overall  throughput  
3  
Obstacle  -  Dense  Environment  
Agents  operate  in  a  warehouse  with  many  blocked  cells  and  narrow  corridors  
Test  pathfinding  quality  and  route  replanning  under  constrained  movement  
4  
Congestion  and  Conflict  Resolution  
Multiple  agents  are  directed  toward  nearby  goals  that  create  traffic  in  shared  routes  
Test  waiting,  avoidance,  and  conflict-resolution  behavior   
5  
Limited  Observability   
Agents  begin  with  only  local  knowledge  and  must  gradually  build  an  internal  map  
Test  partially  observable  decision-making  and  map-based  adaptation  
6  
Dynamic  Change  During  Execution  
A  previously  free  cell  becomes  blocked  or  occupied  while  an  agent  is  en  route  
Test  dynamic  replanning  and  system  robustness  
7  
Allocation  Strategy  Comparison  
The  same  task  set  is  executed  once  with  simple  allocation  and  once  with  auction-based  allocation  
Compare  coordination  strategies  in  terms  of  completion  time,  throughput,  and  congestion  
8  Scalability  Test  
The  number  of  agents  and  tasks  is  gradually  increased  
Evaluate  how  performance  changes  as  system  load  grows  
9  
Temporary  Agent  Trap   
An  agent  becomes  temporarily  trapped  because  nearby  cells  are  occupied  or  blocked  by  other  agents.   
Validate  that  the  agent  does  not  permanently  give  up  when  all  known  paths  appear  blocked.  The  agent  should  continue  waiting,  retrying,  updating  perception,  and  replanning  when  conditions  change.
